## 1. 국내외 유사 서비스 현황

### 1-1. 국내 서비스

#### 같이먹어 (Gati-meogeo)
- **유형**: 공동배달 매칭 앱
- **주요 기능**: 같은 배달 음식을 시킬 사람 모집, 채팅방 생성, 배달비 분할
- **대상**: 1인 가구, 대학가 중심
- **차별점**: 국내 최초 공동배달 특화 앱 중 하나로 알려짐
- **현황**: 서비스 종료 (운영 중단 추정 — 정확한 종료 시점 불확실)
- **플랫폼**: Android / iOS

#### 배달의민족 — 단체주문 기능
- **유형**: 기존 배달 플랫폼 내 부가 기능
- **주요 기능**: 링크 공유로 여러 명이 한 장바구니에 각자 담기
- **차별점**: 이미 대규모 사용자 기반 보유, 결제는 대표자가 처리
- **한계**: 배달비 자동 분할 없음, 낯선 사람과 매칭 기능 없음
- **참고**: 기능 자체는 유사하나 "모르는 사람과의 매칭"은 지원 안 함

#### 쿠팡이츠 — 공유 장바구니 (테스트 중, 2024)
- **유형**: 기존 플랫폼 실험 기능
- **주요 기능**: 지인 초대 후 공동 주문
- **한계**: 지인 기반 (낯선 사람 매칭 없음), 정식 출시 여부 불확실

#### 한솥 (HanSot) — 사내 공동주문 서비스
- **유형**: B2B, 기업 구내식당/사무실 특화
- **주요 기능**: 사무실 단위 공동주문 관리
- **한계**: 학교/일반 소비자 대상 아님

### 1-2. 해외 서비스

#### Sharebite (미국)
- **유형**: 기업 복지형 공동 식사 플랫폼
- **주요 기능**: 팀 단위 음식 주문 조율, 예산 관리, 배달비 절감
- **차별점**: HR/복지 연동, 팀장이 예산 설정 후 팀원이 개별 선택
- **대상**: 기업 직원 (B2B)
- **한계**: 일반 소비자 / 학생 대상 아님
- **출처**: sharebite.com 공식 소개

#### EatWith / Forksy (유럽, 미국)
- **유형**: 소셜 다이닝 — 집밥 공유, 낯선 사람과 식사
- **주요 기능**: 지도 기반 탐색, 호스트가 음식 준비, 게스트 참가
- **차별점**: "함께 먹기" UX — 지도 핀, 프로필, 평점 시스템
- **관련성**: 직접외식 모임 유형 참고 가능
- **출처**: eatwith.com

#### Mealpal (미국, 영국 등)
- **유형**: 구독형 점심 공동구매
- **주요 기능**: 월정액 구독 → 인근 식당 픽업 메뉴를 저렴하게 예약
- **차별점**: 배달 아닌 픽업, 식당과의 대량 계약으로 단가 낮춤
- **성과**: 뉴욕, 런던 등 도심에서 성공적 운영 후 코로나로 사업 축소
- **참고**: 학교 주변 식당 연계 모델로 변형 참고 가능
- **출처**: mealpal.com

#### Squadded (영국, 2019~2021 추정)
- **유형**: 소셜 쇼핑/공동구매 앱 (음식 한정 아님)
- **주요 기능**: 친구와 함께 쇼핑, 의견 교환, 공동 결제
- **차별점**: 채팅 통합 구매 UX
- **현황**: 서비스 종료
- **실패 원인 추정**: 사용자 획득 비용 과다, 네트워크 효과 임계점 미달

#### Mealtrain (미국)
- **유형**: 식사 지원 조율 서비스 (아픈 지인, 출산 등)
- **주요 기능**: 날짜별 식사 배달 담당자 스케줄링
- **관련성**: 낮음 (참고 수준)

## 2. 성공/실패 사례 분석

### 성공 사례

#### Mealpal — 도심 직장인 픽업 모델
- **성공 요인**:
  - 명확한 타겟 (도심 직장인, 점심 한정)
  - 식당과의 B2B 계약으로 구조적 가격 우위 확보
  - 반복 사용 유도 (구독 모델)
- **우리 앱 시사점**: 학교 주변 식당과의 파트너십 → 공동구매 할인 제공 가능성

#### 배달의민족 단체주문
- **성공 요인**:
  - 기존 플랫폼 사용자 기반 활용 (콜드스타트 없음)
  - UX 단순화 (링크 하나로 참여)
- **우리 앱 시사점**: 참여 진입 장벽을 최소화하는 링크/QR 초대 방식 참고

### 실패/중단 사례

#### 같이먹어 (국내)
- **실패 추정 원인**:
  - 네트워크 효과 임계점 도달 실패 — 특정 지역에 충분한 사용자가 없으면 방이 열려도 참여자 없음
  - 결제 정산의 신뢰 문제 (누가 먼저 내는지, 더치페이 분쟁)
  - 배달 시간 동기화 어려움 (각자 생활 패턴 불일치)
- **우리 앱 시사점**: 학교 단위 타겟으로 지역 집중 → 임계점 달성 가능성 높임

#### Squadded (영국)
- **실패 추정 원인**:
  - 사용자 간 네트워크 형성 전에 리텐션 유지 실패
  - 음식 외 범용 공동구매로 집중도 분산
- **우리 앱 시사점**: 초기 타겟을 특정 학교 커뮤니티로 좁혀 네트워크 밀도 확보

## 3. UX / 기능 참고 포인트

### 참여 진입 방식 — 배달의민족 단체주문, EatWith
| 참고 포인트 | 적용 방안 |
|------------|---------|
| 링크/QR 하나로 방 참여 | 카카오톡 공유 → 앱 없어도 웹뷰로 메뉴 선택 가능하게 |
| 지도 핀으로 모임 탐색 | 지도에 현재 열린 방 표시 (공개 방 한정) |

### 신뢰 / 정산 문제 해결 — 국내외 공통 과제
| 참고 포인트 | 적용 방안 |
|------------|---------|
| 선결제 후 참여 구조 | 방 참여 시 개인 몫 선결제 → 주문자 정산 리스크 제거 |
| 평점/매너 온도 (당근마켓 참고) | 함께 주문한 사용자 간 간단 평점 |

### 모임 유형별 UX — EatWith, Mealpal
| 모임 유형 | 참고 서비스 | UX 아이디어 |
|----------|------------|------------|
| 공동배달 | 배민 단체주문 | 장바구니 통합 + 배달비 자동 n분의1 |
| 배달+모여먹기 | 없음 (공백 시장) | 장소 핀 지정 + 인원 제한 설정 |
| 직접외식 | EatWith | 식당 예약 연동 + 참가 확정 버튼 |

### AI 추천 UX — 미적용 서비스 거의 없음 (차별점)
| 기능 | 구현 제안 |
|------|---------|
| 메뉴 추천 | "지금 3명 모임, 예산 15,000원" → 식당+메뉴 콤보 추천 |
| 모임 추천 | 사용자 위치+시간대 기반 "지금 열린 방" 알림 |
| 자연어 대화 | "혼자 먹기 아까운 치킨 같이 시킬 사람?" 입력 → 방 자동 생성 |

위험 요소: 네트워크 효과 임계점 (학교 단위 집중으로 완화 가능)

## 5. 결론 및 추천

1. **직접 경쟁자 없음** — 국내에 "낯선 사람과 실시간 공동배달"을 제대로 서비스하는 앱은 현재 공백 상태. 기회 있음.

2. **과거 실패 서비스의 공통 원인은 네트워크 희소성** — 학교 캠퍼스 단위로 출시 지역을 좁혀 초기 밀도를 확보하는 것이 생존의 핵심.

3. **정산 신뢰 문제를 UX로 해결해야 함** — 선결제 구조 도입이 우선순위. 정산 분쟁 발생 시 서비스 신뢰도가 급락하는 것이 유사 서비스 실패의 공통 패턴.

4. **AI 추천은 실질적 차별점** — 현재 유사 서비스 중 AI 추천을 통합한 사례 없음. 단, 초기 데이터 부족 시 cold start 문제가 있으므로 rule-based 추천과 병행 시작 권장.

5. **EatWith의 "직접외식" UX와 배민 단체주문의 "진입 단순성"** — 두 가지를 결합한 설계가 우리 앱의 UX 방향으로 가장 적합.

**출처 및 신뢰도**
- 배달의민족 단체주문: 공식 서비스 직접 확인 (고신뢰)
- Mealpal, EatWith, Sharebite: 공식 웹사이트 기반 (고신뢰)
- 같이먹어 서비스 종료 여부: 앱스토어 미등록 확인 기반, 종료 시점은 불확실 (중신뢰)
- Squadded 실패 원인: 공개 보도 기반 추정 (중신뢰, 내부 사정 불확실)
